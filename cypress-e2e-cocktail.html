Cypress E2E Suite — Cocktail Finder

This project is a Cypress end-to-end (E2E) UI testing suite for my Cocktail Finder web app. It exercises the application the way a real user would: loading random cocktails, running searches, navigating between views, and saving or removing favorites backed by localStorage.

Paired with my API Testing Suite for TheCocktailDB, it provides layered coverage across both the API and the browser experience that consumes it. The Cypress suite runs in real browsers locally and is wired into GitHub Actions, so headless E2E checks can run automatically on every push.

Project Goals & Context

Cocktail Finder is a single-page app that talks to TheCocktailDB and lets users discover new drinks, search for specific cocktails, and save favorites for later. As I iterated on the UI, small changes to layout, state handling, or API usage could easily break pieces of the experience in ways that weren’t obvious from API tests alone.

I created this Cypress suite to give myself a reliable, repeatable way to prove that the main flows still work after refactors, styling updates, and new features. Instead of relying on ad-hoc manual checks, I wanted automated tests that represented how real users move through the app in an actual browser.

My goals for this project were:

  • Validate the most important user journeys end-to-end in a real browser (random cocktails, search, navigation, and favorites).
  • Keep UI state and localStorage behavior in sync across reloads and sessions.
  • Use maintainable automation patterns — explicit data-testid selectors, fixtures, and custom commands — so tests stay readable and resilient as the UI changes.
  • Exercise flows across multiple viewport sizes and browsers when running locally with the Cypress Test Runner.
  • Make the suite CI-friendly so headless E2E checks can run automatically in GitHub Actions on every push.

Test Coverage Overview

The suite focuses on the flows that matter most to someone using Cocktail Finder: loading random drinks, running searches, and managing favorites.

Random Cocktail Flow

  • Loads the homepage and verifies an initial random cocktail is displayed.
  • Clicks the “random” button to fetch a new drink and checks that details update.
  • Asserts core UI elements (name, image, ingredients, instructions) are present.
  • Gracefully handles temporary API errors with a friendly error message.

Search & Results

  • Search by full cocktail name (e.g., margarita).
  • Search by partial name and verify multiple results render correctly.
  • Search with no matches and confirm an appropriate “no results” UI.
  • Variations for uppercase/lowercase to ensure case-insensitive matching.

Favorites & localStorage

  • Favorite a cocktail from search or random results.
  • Verify that the Favorites section updates immediately.
  • Reload the page and ensure favorites persist via localStorage.
  • Unfavorite a drink and confirm it is removed from both UI and storage.

Navigation & Layout

  • Check that key navigation controls and filters are accessible.
  • Basic responsiveness checks with multiple viewports (desktop and mobile widths).
  • Ensure focus states and keyboard navigation work for core controls.

Automation Architecture

The Cypress project uses a standard modern layout with E2E specs, shared helpers, and fixtures.

Tools & Stack

  • Cypress for browser-based E2E testing.
  • JavaScript test specs and helpers.
  • Fixtures for sample search queries and mock drink data.
  • Custom commands to log steps like favoriting and verifying items.
  • Node.js + npm for dependencies and scripts.

Project Structure

  • cypress/e2e/ — feature-based specs (random, search, favorites, navigation).
  • cypress/fixtures/ — reusable data for search terms and sample drinks.
  • cypress/support/commands.js — custom commands like cy.addFavorite().
  • cypress.config.js — base URL and global configuration.

Selector Strategy & Reusable Commands

To keep the tests stable as the UI evolves, I rely on explicit data-testid attributes for key elements instead of fragile CSS selectors.

  • Buttons like data-testid="random-btn" and data-testid="favorite-btn".
  • Search input with data-testid="search-input".
  • Result cards with data-testid="drink-card".

I also added custom commands for common flows:

  • cy.visitApp() — opens the Cocktail Finder app at its base URL.
  • cy.searchFor('margarita') — types into the search box and submits.
  • cy.addFavorite('Margarita') — favorites a card by drink name.

These commands keep the specs readable and make it easy to reuse patterns across tests.

CI Integration with GitHub Actions

The suite is wired to run automatically in GitHub Actions so E2E checks happen on each push.

Workflow Highlights

  • Installs Node.js and Cypress in a CI environment.
  • Runs npm run test:e2e (Cypress headless mode).
  • Surfaces pass/fail status and captured screenshots/videos on failure.

Why It Matters

  • Protects the user experience when I refactor UI code.
  • Shows hands-on experience using Cypress in a CI pipeline.
  • Pairs nicely with the Postman/Newman API suite to provide full-stack coverage.

How to Run the Suite Locally

Running the Cypress suite only requires Node.js and npm:

  1. Clone the repository: git clone https://github.com/heathergauthier2018/cocktail-finder-cypress-e2e.git
  2. Install dependencies: npm install
  3. Run tests in the Cypress GUI: npx cypress open
  4. Or run headless (useful for CI): npx cypress run

The project’s cypress.config.js file points to the deployed Cocktail Finder app, so tests run against the same environment users see.

Reflection

This project strengthened my end-to-end testing mindset by pushing me to think beyond API behavior and validate how real users experience the application. I learned to view UI flows as contracts in their own right—contracts between the interface, the underlying state, and the API responses that shape what the user sees.

Cypress helped me practice automation patterns used widely on QA and SDET teams: resilient selector strategies, custom reusable commands, viewport and cross-browser testing, and CI-driven E2E validation. Designing tests this way required me to think critically about state, timing, accessibility, and user intent—not just what appears on screen.

Going forward, I can reuse this approach for any front-end application I test. Whether I’m validating my own projects or working within a larger engineering team, this Cypress suite gives me a solid foundation in modern UI automation, reliable regression coverage, and clean, maintainable test design.
